# Telegram Bot для сбора сообщений об ошибках сервиса проверки ВКР

Telegram-бот предназначен для регистрации, обработки и сопровождения сообщений об ошибках, возникающих при работе сервиса проверки оформления выпускных квалификационных работ (ВКР).

Основные возможности:

- регистрация новых сообщений об ошибках;
- загрузка отчёта пользователя (.docx/.txt);
- назначение обращения администратору;
- сопровождение обращения до закрытия;
- хранение полной истории изменений;
- автоматическая проверка качества описания ошибки при помощи модели на основе цепей Маркова;
- возможность дообучения модели непосредственно администраторами.

# Архитектура проекта

Проект разделён на несколько независимых модулей.

```
project/

├── keyboards/          # клавиатуры
├── lexicon/            # локализация
├── middlewares/        # middleware
├── services/           # бизнес-логика
├── logs/               # журналы
└── main.py             # точка входа
```

# Структура database

```
database/
│
├──db/
|  ├── create.py              # создание записей
|  ├── update.py              # изменение записей
|  ├── queries.py             # получение отдельных объектов
|  ├── pagination.py          # получение страниц
|  ├── counters.py            # функции подсчёта
|  ├── common.py              # общие структуры и SQL
├── models.py                 # ORM модели
└── connection.py             # создание подключения
```

## Почему функции разделены

Вместо одного большого файла используются специализированные модули.

Например

```
create.py
```

отвечает исключительно за создание данных.

```
update.py
```

изменяет существующие записи.

```
queries.py
```

получает отдельные объекты.

```
pagination.py
```

возвращает страницы результатов.

Такой подход уменьшает связанность кода и облегчает сопровождение.

# Архитектура хранения багов

Каждый баг состоит из трёх сущностей.

## BugReport

Основная карточка обращения.

Содержит постоянную информацию:

- автор
- название
- дата создания

---

## BugData

Версия обращения.

При каждом повторном открытии создаётся **новая версия**, поэтому история полностью сохраняется.

Хранятся:

- описание
- файл отчёта
- номер версии
- признак использования в обучении модели

---

## BugStatus

История изменения статусов.

Каждое изменение создаёт новую запись.

Хранятся:

- статус
- критичность
- назначенный администратор
- время изменения

Благодаря этому можно восстановить состояние обращения на любом этапе.

---

# BugView

Во всех обработчиках используется объект

```
BugView
```

Это не таблица базы данных.

Это объединение данных из:

- BugReport
- BugData
- BugStatus

Он избавляет остальные части программы от необходимости самостоятельно выполнять несколько SQL-запросов.

---

# Модель проверки описаний

В проект встроена собственная модель проверки качества описаний багов.

Она реализована без использования библиотек машинного обучения.

Используются:

- цепи Маркова второго порядка;
- резервная модель первого порядка;
- вероятностная оценка текста.

Модель определяет, похоже ли описание пользователя на реальные сообщения об ошибках.

Если описание признано слишком коротким или бессмысленным, пользователю предлагается переписать его.

# Обучение модели

Источники обучения:

- файл

```
config/data/train.txt
```

- записи из базы данных, помеченные администраторами.

Для обучения используется поле

```
BugData.is_training_sample
```

Только записи с этим флагом участвуют в обучении.

---

# Дообучение модели

Администратор может добавить любую версию обращения в обучающую выборку.

После каждого изменения счётчик обучения увеличивается.

Когда количество изменений достигает значения

```
BUG_DESCRIPTION_MODEL_RETRAIN_AFTER_CHANGES
```

происходит автоматическое переобучение модели.

После успешного обучения счётчик автоматически сбрасывается.

---

# Проверка описания

Во время создания обращения выполняется

```
MarkovModel.validate(...)
```

Если модель считает описание некорректным:

- пользователь получает сообщение;
- состояние FSM не меняется;
- описание необходимо отправить повторно.

# Логирование

Проект использует два независимых журнала.

## Основной журнал

```
logs/bot.log
```

Содержит:

- запуск бота;
- ошибки;
- регистрацию обращений;
- действия администраторов.

---

## Журнал отклонённых описаний

```
logs/invalid_descriptions.log
```

Записывается независимо от уровня логирования.

Содержит:

- время;
- id пользователя;
- текст отклонённого описания.

Используется для анализа качества модели.

---

# FSM

Создание обращения реализовано через FSM.

Последовательность состояний:

```
ожидание описания
        ↓
ожидание файла
        ↓
создание обращения
```

При отмене состояние полностью очищается.

---

# Роли

В системе существуют три роли.

## Пользователь

Может:

- создать обращение;
- повторно открыть обращение;
- подтвердить исправление.

---

## Администратор

Может:

- принять обращение;
- установить критичность;
- завершить исправление;
- пометить описание как некорректное;
- добавить пример в обучающую выборку.

---

## Суперадминистратор

Дополнительно может:

- добавлять администраторов;
- удалять администраторов.

---

# Конфигурация

Основные параметры модели:

```
BUG_DESCRIPTION_MODEL_ENABLED
```

Включение проверки описаний.

```
BUG_DESCRIPTION_MODEL_BYPASS
```

Пропуск проверки (модель считается включённой, но принимает любые описания).

```
BUG_DESCRIPTION_MODEL_SECOND_ORDER_THRESHOLD
```

Порог модели второго порядка.

```
BUG_DESCRIPTION_MODEL_FIRST_ORDER_THRESHOLD
```

Порог резервной модели.

```
BUG_DESCRIPTION_MODEL_FIRST_ORDER
```

Использовать ли резервную модель.

```
BUG_DESCRIPTION_MODEL_RETRAIN_ON_START
```

Переобучать модель при запуске.

```
BUG_DESCRIPTION_MODEL_RETRAIN_AFTER_CHANGES
```

Через сколько изменений выполнять автоматическое переобучение.

# История обращений

Каждое изменение обращения создаёт новую запись в BugData.

Поэтому можно просмотреть:

- первоначальное описание;
- все последующие исправления;
- прикреплённые файлы каждой версии.

История никогда не удаляется.
